Introduction to Out-of-Order Processors

Posted on 2019-09-22

Life-Cycle of An Instruction

  1. Instruction Fetch: the instruction is fetched from memory or an instruction cache.
  2. Issue: the instruction is renamed.
  3. Register Fetch: The instruction waits, if necessary, in a resevation station until its source operands have valid status and a suitable functional unit is available.
  4. Execution: when all source operands are ready and a suitable functional unit is available the execution begins. Different operations can have different latencies.
  5. Write Back: when the assigned functional unit completes its execution, it writes the destination operand value to the designated value register.
  6. Commitment: the value register that was previously assigned to the instruction’s destination architectural register can be freed for reuse by another instruction.

Speculation and Branches

The Motivation of Branch Prediction

Branch penalties limit performance of deeply pipelined processors. Modern branch predictors have high accuracy(>95%) and can reduce branch penalties significantly.

Required Support of Branch Prediction

  1. Prediction structures: branch history tables, branch target buffers, etc.
  2. Mispredict recovery mechanisms:
    • In-order machines: kill instructions following branch in pipeline.
    • Out-of-order machines: shadow registers and memory buffers for each speculated branch.

Dynamic Branch Prediction: Learning Based on Past Behavior

  1. Temporal Correlation: the way a branch resolves may be a good predictor of the way it will resolve at the next execution.
  2. Spatial Correlation: several branches may resolve in a highly correlated manner.

WAW and WAR Name Dependency

  1. WAW and WAR are not true data dependencies.
  2. RAW is true data dependency because reader truely needs result of writer.
  3. WAW and WAR are name dependencies, which exist because we have limited number of names(register specifiers or memory addresses)

Register Renaming

Reference: https://www.d.umn.edu/~gshute/arch/register-renaming.xhtml
Register renaming is a form of pipeline that deal with data dependencies between instructions by renaming their register operands. An assembly language programmer or a compiler specifies these operands using architectural registers, which are registers that are explicit in the instruction set architecture. Renaming replaces architectural names by value names with a new value name for each instruction destination operand. This eliminates the name dependencies between instructions and automatically recongnized true dependencies.

Benefits Brought By Renaming

With the renaming strategy, WAR and WAW, not true data dependencies, can be eliminated.

What problems exist with adding infinite number of registers to a computer instruction set architecture?
It takes up encoding space and encoding space is quite limited.

Hardware Implementation of Renaming

  1. Instruction Fetch: fetch instructions from memory or an instruction cache.
  2. Issue: rename instructions.
  3. Reservation: holds renamed instructions until they are ready to be executed.
  4. Execution: perform operations required for instructions.
  5. Values: manages a numbered set of value registers. The register number are used by issue circuitry for renaming operands. The value registers can also named as rename registers
  6. Commit: free value register that are no longer needed.

Memory Disambiguation

Give a sequence of instructions:

  1. st R1, 0(R2)

In-Order Memory Queue

  1. Execute all loads and stores in program order, which means load and store cannot leave IQ for execution until all previous loads and stores have completed execution.

Conservative OOO Load Execution

  1. Split execution of store instruction into two phases: address calculation and data write.
  2. Can execute load before store, if addresses know and R4 != R2.
  3. Each load address compared with addresses of all previous uncommitted stores
  4. Don’t execute load if any previous store address not known

Memory Dependence Prediction

  1. Guess that R4 != R2 and execute load before store
  2. If later find that R4 == R2, squash load and all following instructions, but mark load instruction as store-wait.
  3. Subsequent executions of the same load instruction will wait for all previous stores to complete.
  4. Periodically clear store-wait bits.

Limits of Out-of-Order Design Complexity

Firstly, give a figure to elaborate the superscalar control logic:

We can observe that:

  1. Each issued instruction must somehow check against $W*L$ instructions, i.e., growth in hardware in O(WWL)
  2. For in-order machines, L is related to pipeline latencies and check is done during issue.
  3. For out-of-order machines, L also includes time spent in IQ, SB and check is done by broadcasting tags to waiting instructions at completion.
  4. As W increases, larger instructions window is needed to find enough parallelism to keep machine busy ==> greater L. As you go wider, your pipeline needs to get deeper.

All of these mean that out-of-order control logic grows faster than $W^2$, almost $W^3$.
